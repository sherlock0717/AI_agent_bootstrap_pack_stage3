## A. AI 综合生成（策略驱动能力调用）

## 目标对齐
基于现有材料，《梦土回声》项目当前核心目标为：**将已形成完整创意资产（世界观、角色、Demo剧本）的项目，系统化接入AI协作流程，稳定推进首个高密度、可试玩Demo的制作与验证**。所有后续工作必须围绕“Demo主链”展开，优先保障玩家在前15分钟内的体验成立（识别独特气质、记住角色与场景、愿意继续游玩），并确保项目在系统化过程中不丧失其“诗性、都市神秘、情感残响”的创作核心。

## 里程碑与优先级（用 P0/P1/P2）
- **P0（必须立即执行）**：
  - 按《首次接手执行顺序》完成项目入口文件阅读（README， Manifest， INDEX）。
  - 建立项目专属索引页与Demo主链状态页，明确冻结与可调整内容。
  - 生成首轮最短闭环任务单（3-5个关键任务）。
  - 完成“工作包A：项目冻结与知识入库”，将核心设定、角色卡等转化为系统可引用的知识层。

- **P1（近期重点推进）**：
  - 推进“工作包B：Demo剧本制作包整理”，将剧本拆分为可供制作使用的结构化脚本、场景表和变量表。
  - 启动“工作包C：UI与演出规格包”与“工作包D：美术需求包”的规格制定，产出可执行的UI Spec和美术资源清单。
  - 依据系统接入路线图，完成“Demo执行主链接入”，明确当前可继续推进的P0工作项。

- **P2（后续规划）**：
  - 推进“工作包E：音频需求包”和“工作包F：实现与验证包”的准备工作。
  - 建立“工作包G：项目复盘与系统回写”的模板与流程。
  - 准备首轮试玩与后续修订闭环。

## 依赖与风险
- **关键依赖**：
  - **人工决策**：母命题、核心角色性格、主线方向、关键视觉气质等创作核心判断必须保留人工主导。
  - **资产就绪**：Demo实现依赖于剧本包、UI/美术/音频等规格包的先行完成与确认。
  - **系统对接**：需按《请先看》说明，正确将外部项目目录与AI系统知识库、交付物目录对接。

- **主要风险**：
  - **体验稀释**：过度系统化、流程化可能导致项目独特的“角色温度”与“情感残响”气质被削弱。
  - **范围蔓延**：偏离“Demo主链”，过早进行世界观扩写、新增角色或复杂玩法扩建。
  - **接手偏差**：AI系统未遵循“先读冻结层”原则，擅自改动高层设定或重开议题。
  - **资产瓶颈**：美术、音频等资源的生产进度可能影响Demo原型实现与试玩节点。

## 近期可执行动作（条目标注责任人占位如「程序/美术/叙事」若材料中有）
1.  **责任人：系统/项目负责人**：按照《请先看》说明，将外部项目目录（`DreamEcho_Demo`）中的关键入口文件（README, INDEX, Manifest等）同步至AI系统对应的knowledge与deliverables目录，完成项目初始落仓。
2.  **责任人：系统/叙事**：执行“工作包A”，将已冻结的核心世界观、角色卡、第一季章节卡、Demo剧本等整理并录入系统知识库（`knowledge/`目录下），生成固定术语表与风格禁区表。
3.  **责任人：系统/叙事**：创建《梦土回声》项目专属索引页与Demo主链状态页，清晰标注各文件链的冻结状态（冻结/可调整/待补充）。
4.  **责任人：系统/叙事**：依据“工作包B”要求，开始将第一章及第二章前半的锁定剧本稿，拆分为结构化的脚本文件、场景表（CSV）和分支变量表（CSV）。
5.  **责任人：美术/系统**：依据“工作包D”要求，开始整理Demo所需的背景、立绘、CG清单，形成附有关键词与禁区的美术需求文档（`demo_art_request.md`）及资源清单CSV。
6.  **责任人：程序/美术/系统**：启动“工作包C”，协同制定UI与演出规格草案，包括对话框、选择框、异常状态特效等（产出`demo_ui_spec.md`草案）。
7.  **责任人：项目负责人/系统**：根据以上动作的输出，生成一份包含3-5个任务的《首轮闭环任务单》，明确下一步具体的执行内容、交付物及验收标准。

# 《梦土回声》AI 系统接手顺序与执行路线图（含双轮审阅修订版 V1）

## 一、总原则

AI 系统接手《梦土回声》时，必须遵循以下总原则：

1. **先接 Demo 主线，不先扩世界**
2. **先读冻结层，不先改写高层设定**
3. **先推进可执行项，不先生成更多泛模板**
4. **所有后续动作都要回到玩家体验，而不是只优化流程**

---

## 二、接手顺序总览

AI 系统接手顺序建议固定为 5 步。

### 第一步：项目定位复核
目标：确认系统知道《梦土回声》当前不是在做“新 IP 立项”，而是在推进“首发 Demo 主链”。

应先读：
- 最终总交付总说明
- 核心设定与第一季开发骨架
- Demo brief

输出应为：
- 当前项目阶段判断
- 当前主目标判断
- 明确不做事项清单

### 第二步：冻结边界复核
目标：确认系统知道哪些内容不能乱改。

应先读：
- 后续迭代口径与冻结边界
- 第一批核心角色正式角色卡
- 第一季五章章节卡
- 第 2 周锁定产物包

输出应为：
- 冻结层确认单
- 可调整层确认单
- 延后层确认单

### 第三步：Demo 执行主链接入
目标：让系统真正接入当前最该继续推进的资料链。

应先读：
- Demo spec
- Ren’Py 实现包
- 视觉风格圣经
- 音频与配音演出圣经
- 第 4 周执行版文件

输出应为：

- 当前不建议推进的 P1 / P2 工作项

### 第四步：评审与修订链接入
目标：让系统知道后续不是继续无边扩写，而是要围绕评审、试玩、修订闭环推进。

应先读：
- 评审标准包
- 评审结论模板包
- 修改指令单包
- 首轮试玩执行包
- 二次修订示例稿

输出应为：
- 评审回路说明
- 修订闭环说明

### 第五步：系统化持续推进
目标：在不破坏项目边界的前提下，进入可持续协作。

系统后续可做：
- 继续细化 Demo 执行层
- 协助生成评审稿、回传稿、试玩总结稿
- 协助归档、版本说明、对外展示更新

系统后续暂不做：
- 第二季扩写
- 大规模番外
- 新角色群大扩编
- 重做世界异常逻辑
- 把项目转型成高成本动作玩法产品

## 三、系统接手后的首轮执行建议

AI 系统正式接手后的第一轮执行，建议只做以下三件事：

### 1. 建立《梦土回声》项目专属索引页
让后续所有推进都从统一目录进入。

### 2. 建立 Demo 主链状态页
明确当前哪些文件已经冻结，哪些还在等待资产回填或实机验证。

### 3. 生成下一轮最短闭环任务单
只围绕 Demo 当前最关键的 3—5 个任务继续推进。

## 四、接手后的禁止动作

以下动作在系统接手后的前两轮推进中应明确禁止：

1. 重新写一版世界观总稿
2. 重构主要角色功能定位
3. 重写 Demo 的前 15 分钟抓手
4. 大量补充与 Demo 当前无关的世界扩展设定
5. 继续生成泛用流程模板，替代当前已经够用的文件链
